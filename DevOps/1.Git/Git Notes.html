What is Git?

The Big Picture

Git is a distributed version control system (DVCS) that tracks changes in files over time. Every developer has a complete copy of the entire project history on their machine — not just the latest snapshot. This means you can work offline, create branches freely, and collaborate with anyone.

📸 Snapshots, not diffs

Git stores snapshots of your entire project at each commit, not just the lines that changed. This makes switching branches and reverting instant.

🌿 Branching is cheap

Creating a branch in Git is almost instant — it's just a pointer to a commit. You can have dozens of experimental branches with zero performance cost.

🌐 Distributed

Every clone is a full backup. There's no single point of failure. You can push to GitHub, BitBucket, GitLab, or your own server.

🔐 Data integrity

Every commit, file, and tree object is identified by a SHA-1 hash. It's physically impossible to change history without Git detecting it.

Git vs Other VCS

FeatureSVN / CVS (Centralized)Git (Distributed)
Repository locationOne central serverEvery clone is a full repo
Offline work❌ Need server connection✅ Full history locally
Branching costExpensive, slowNear-zero cost
SpeedNetwork-dependentMostly local operations
History modelLinear diffsDAG of snapshots

The Three States of Git

Every file in your Git project lives in one of three states. Understanding this is the foundation of everything else in Git.

Git's Three-State Model
📁 Working Directory
Files you can see & edit
──git add──▶
📋 Staging Area (Index)
Prepared for next commit
──git commit──▶
🗃️ Repository (.git)
Permanent snapshot history
💡
Why a Staging Area?

The staging area lets you craft precise commits. You might edit 5 files but only stage 2 of them — creating a focused, meaningful commit rather than dumping all changes at once.

Installing & Configuring Git

Installation

🖥️ Install Git by OS
# macOS — via Homebrew
brew install git

# Ubuntu / Debian Linux
sudo apt install git

# Windows — download installer at:
# https://git-scm.com/download/win

# Verify install
git --version
# → git version 2.44.0

First-time Configuration

Git has three levels of configuration: system (all users), global (your user account), and local (this repo only). The most important is global — set your name and email before making your first commit.

⚙️ git config — Identity Setup
# Set your name (appears in every commit you make)
git config --global user.name "Your Name"

# Set your email (must match GitHub for credit)
git config --global user.email "you@example.com"

# Set default editor (VS Code)
git config --global core.editor "code --wait"

# Set default branch name to 'main' (modern standard)
git config --global init.defaultBranch main

# View all your settings
git config --list
🎨 Useful Aliases (shortcuts)
# Create shortcuts for common commands
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.lg "log --oneline --graph --decorate --all"

# Now you can type:
git st        # instead of: git status
git lg        # beautiful one-line graph log
✅
Getting Help

Run git help <command> or git <command> --help to open the full manual page. For a quick summary of flags, use git <command> -h.

03

Git Workflow — Initializing a Repository

Initializing a Repo

📂 git init
# Create a new folder and initialize Git inside it
mkdir my-project
cd my-project
git init

# Git creates a hidden .git/ folder here — this IS the repository
# Your working directory is just the files around it

ls -la
# drwxr-xr-x  .git/   ← The entire repo lives here
# (no other files yet)

The Full Workflow Loop

Day-to-day Git Workflow
Edit files → git add <file> → git commit -m "msg"
    ↕ modify more          ↕ stage more files          ↕ repeat
Tip: One logical change = one commit. Don't bundle unrelated work.

Commit Best Practices

  • One logical change per commit — fix a bug OR add a feature, not both
  • Present-tense, imperative message — "Add login form" not "Added login form"
  • Short subject line (≤ 50 chars) — GitHub truncates at 72
  • Don't commit broken code — your commits should always compile / pass tests
  • Avoid "WIP" commits — use stash or branches for work in progress
✅ Good vs ❌ Bad Commit Messages
# ❌ Bad — too vague, past tense, random bundling
git commit -m "fixed stuff and added things"

# ✅ Good — specific, imperative, one concern
git commit -m "Fix null pointer in UserService.login()"
git commit -m "Add password strength indicator to signup form"
git commit -m "Refactor: extract parseDate() helper"

Staging & Committing

Essential Commands

📋 Staging files
# Stage a single file
git add index.html

# Stage multiple files
git add style.css app.js

# Stage all changes in current directory (modified + new)
git add .

# Stage all tracked modified files (skip untracked)
git add -u

# Stage parts of a file interactively (hunk by hunk)
git add -p index.html
💾 Committing
# Commit with inline message
git commit -m "Add hero section to homepage"

# Stage ALL tracked files AND commit in one step (skip staging)
git commit -am "Fix typo in README"
# ⚠️  Does NOT include brand-new (untracked) files!

# Open editor for a longer commit message
git commit
🗑️ Removing & Renaming files
# Remove file from repo AND disk
git rm old-file.txt

# Remove from repo but KEEP on disk (untrack it)
git rm --cached secret.env

# Rename or move (Git detects the rename automatically)
git mv old-name.js new-name.js

Checking Status

🔍 git status
# Full status output
git status

# Short / compact output
git status -s
# Output format: XY filename
# X = staging area status   Y = working directory status
# M = modified  A = added  D = deleted  ? = untracked

# Example output:
# M  src/app.js      ← staged modification
#  M src/style.css   ← unstaged modification
# ?? new-file.txt    ← untracked file

Viewing Changes (Diff)

git diff — What Changed?

🔎 git diff commands
# Unstaged changes (working dir vs staging area)
git diff

# Staged changes (staging area vs last commit)
git diff --staged
# alias: git diff --cached  (same thing)

# Compare two commits
git diff abc1234..def5678

# See what changed in a specific file between commits
git diff HEAD~2 HEAD -- src/app.js

# Reading diff output:
# --- a/file.js   ← old version
# +++ b/file.js   ← new version
# -removed line   ← red, was there before
# +added line     ← green, new addition
💡
Visual Diff Tools

For a nicer view, configure VS Code as your diff tool: git config --global diff.tool vscode. Then run git difftool to open the visual side-by-side comparison.

Ignoring Files — .gitignore

What to Ignore

Some files should never be committed: secrets, build artifacts, OS junk, and dependency folders. The .gitignore file tells Git to completely ignore matching paths.

.gitignore — Real-world Example
# Dependencies
node_modules/
vendor/

# Build output
dist/
build/
*.min.js

# Environment / secrets
.env
.env.local
*.pem
*.key

# OS files
.DS_Store       # macOS
Thumbs.db       # Windows

# IDE files
.vscode/settings.json
.idea/

# Logs
*.log
logs/
🔤 .gitignore Pattern Syntax
# Ignore all .log files anywhere in the project
*.log

# Ignore only the root-level debug.log, not sub/debug.log
/debug.log

# Ignore a whole directory
node_modules/

# Negate (un-ignore) a pattern — KEEP this file despite *.log above
!important.log

# Ignore all .txt files in any 'docs' subdirectory
docs/**/*.txt
⚠️
Already Tracked Files

Adding a file to .gitignore won't stop tracking it if Git is already tracking it. You must run git rm --cached filename first to untrack it, then the .gitignore rule kicks in.

07

Undoing Changes

Un-staging, Discarding & Restoring

↩️ Undoing at each stage
# 1. Unstage a file (move it back from staging area to working dir)
git restore --staged file.js
# Old syntax: git reset HEAD file.js

# 2. Discard working directory changes (DANGEROUS — cannot undo!)
git restore file.js
# Throws away ALL uncommitted changes to file.js

# 3. Restore a file to a specific commit's version
git restore --source=abc1234 file.js
# The file now looks like it did at that commit
Where to use which undo command
Working Dir → git restore file ← discards edits
Staging Area → git restore --staged file ← back to working dir
Committed → git revert / git reset ← undo commit
🚫
git restore is irreversible for working directory changes

Once you run git restore file.js, the edits in your working directory are gone forever (unless you have a backup). There is no undo for discarding uncommitted changes.

Filtering & Formatting History

Powerful Log Filters

🔎 git log filtering options
# By author
git log --author="Alice"

# By date range
git log --after="2025-01-01" --before="2025-04-01"

# By keyword in commit message
git log --grep="login"

# Commits that changed a specific string in the code
git log -S "parseDate"

# History of a specific file
git log --oneline -- src/utils.js

# Commits that touched a specific function
git log -L :parseDate:src/utils.js

Custom Log Formatting

🎨 --pretty=format
# Custom one-line format with color
git log --pretty=format:"%Cgreen%h%Creset %Cblue(%ar)%Creset %s — %an"

# Format tokens:
# %h = short hash    %H = full hash
# %s = subject       %b = body
# %an = author name  %ae = author email
# %ar = relative date ("3 days ago")
# %ad = absolute date

Checking Out Commits & Tagging

Checking Out a Past Commit

You can "travel back in time" to inspect any previous snapshot. This puts Git into detached HEAD state — you're not on any branch, just viewing a commit directly.

⏳ git checkout — time travel
# Go back to a specific commit (detached HEAD)
git checkout abc1234

# You'll see: "HEAD detached at abc1234"
# Browse code, run it, copy files — but don't make commits here

# Return to the latest commit on your branch
git checkout main
# or: git switch -

# Relative references
git checkout HEAD~1   # one commit before HEAD
git checkout HEAD~3   # three commits before HEAD

Tags — Permanent Bookmarks

Tags mark important points in history — usually releases. Unlike branch pointers, tags don't move when you commit.

🏷️ git tag
# Lightweight tag (just a pointer, like a bookmark)
git tag v1.0

# Annotated tag (recommended — has tagger name, date, message)
git tag -a v1.0 -m "Version 1.0 — first stable release"

# Tag a specific past commit
git tag -a v0.9 abc1234 -m "Beta release"

# List all tags
git tag

# Delete a tag
git tag -d v0.9

# Push a tag to remote
git push origin v1.0

# Push ALL tags
git push origin --tags

Bisect & Blame — Finding Bugs

git bisect — Binary Search for Bugs

When you know a bug exists now but didn't exist at a past commit, git bisect performs a binary search through history — cutting the search space in half each time to pinpoint exactly which commit introduced the bug.

🔍 git bisect workflow
# 1. Start bisect session
git bisect start

# 2. Mark current state as broken
git bisect bad

# 3. Mark a known-good past commit
git bisect good v1.0
# Git jumps to the midpoint commit for you to test

# 4. Test the code — then tell Git the result
git bisect good    # if this commit is fine
git bisect bad     # if this commit is broken
# Repeat until Git says: "abc1234 is the first bad commit"

# 5. End the session
git bisect reset

git blame — Who Wrote This Line?

👤 git blame
# Show every line of a file annotated with who last changed it
git blame src/app.js

# Output: hash | author | date | line number | content
a3f91bc4 (Alice Smith  2025-03-14 09:21) 42 | function parseDate(str) {
b2e7fd12 (Bob Johnson  2025-02-10 14:05) 43 |   return new Date(str);
a3f91bc4 (Alice Smith  2025-03-14 09:21) 44 | }

# Blame only specific lines
git blame -L 40,50 src/app.js

# VSCode GitLens extension shows blame inline as you type

Working with Branches

What is a Branch?

A branch is just a lightweight movable pointer to a commit. When you create a branch, Git creates a new pointer — not a copy of files. This is why branching is nearly instant in Git.

Branch Model — main & feature branch
C1 → C2 → C3 → C4 ← main, HEAD
↘
C3a → C3b ← feature/login
🌿 Branch Commands
# List all branches (* marks current)
git branch

# Create a new branch (but don't switch)
git branch feature/login

# Create AND switch to new branch (modern syntax)
git switch -c feature/login
# Old syntax: git checkout -b feature/login

# Switch to existing branch
git switch main

# Rename current branch
git branch -m new-name

# Delete a branch (must be fully merged)
git branch -d feature/login

# Force delete (even if not merged)
git branch -D feature/login

# Compare branches — commits in feature not in main
git log main..feature/login --oneline

Stashing — Save Work Without Committing

Need to switch branches but have uncommitted work you're not ready to commit? Stash it!

📦 git stash
# Save current uncommitted changes to a stash
git stash

# Stash with a descriptive message
git stash push -m "WIP: half-done search feature"

# List all stashes
git stash list
# stash@{0}: WIP: half-done search feature
# stash@{1}: On main: another saved state

# Apply most recent stash AND remove it
git stash pop

# Apply a specific stash WITHOUT removing it
git stash apply stash@{1}

# Drop (delete) a stash
git stash drop stash@{0}

Merging — Combining Branches

Fast-Forward Merge

When the branch being merged has no competing changes — the base branch hasn't moved since the feature branch was created — Git simply moves the pointer forward. No merge commit is created.

Fast-Forward Merge
Before: C1 → C2 ← main      C3 → C4 ← feature
After:  C1 → C2 → C3 → C4 ← main (moved forward)
⚡ Fast-forward merge
# Switch to the target branch (the one you're merging INTO)
git switch main

# Merge the feature branch (fast-forward if possible)
git merge feature/login

# Disable fast-forward — always create a merge commit
git merge --no-ff feature/login

# After merge, clean up the feature branch
git branch -d feature/login

Three-Way Merge

When both branches have diverged (both have new commits since they split), Git performs a three-way merge: it looks at both branch tips AND their common ancestor to create a new merge commit.

Three-Way Merge
C1 → C2 → C3 ← main
↘
C2a → C2b ← feature
Result: C1 → C2 → C3 → M ← merge commit
↗ (merges C2a + C2b)

Squash Merge

Squash merge combines all commits from the feature branch into a single new commit on main — keeping main's history clean and linear.

🗜️ Squash merge
# Squash all feature commits into one staged change
git merge --squash feature/search
# Changes are now staged, but NOT yet committed

# Write a single clean commit message
git commit -m "Add search feature"

# Delete the feature branch (it's been absorbed)
git branch -D feature/search
14

Merge Conflicts — Resolving Disagreements

When Conflicts Happen

A conflict occurs when the same lines in the same file were changed differently on both branches. Git can't decide which version to keep — it asks you to resolve it.

⚔️ What a conflict looks like
<<<<<<< HEAD (main)
const greeting = "Hello, World!";
=======
const greeting = "Hi there, World!";
>>>>>>> feature/greet

# The section between <<<< and ==== is YOUR branch (HEAD)
# The section between ==== and >>>> is the INCOMING branch
# You must manually edit the file to the correct final state
✅ Resolving the conflict
# Step 1: Open the file and edit to the desired final state
# Delete all the conflict markers, keep what you want:
const greeting = "Hi there, World!"; // ← chosen version

# Step 2: Stage the resolved file
git add src/app.js

# Step 3: Complete the merge
git commit

# Alternative: abort the merge entirely
git merge --abort
✅
Use VS Code to Resolve Conflicts

VS Code shows conflicts with "Accept Current Change / Accept Incoming Change / Accept Both" buttons inline. This is much easier than editing raw conflict markers.

Rebasing — Rewrite History Cleanly

What is Rebase?

Rebasing re-applies your commits on top of another branch — as if you had started your feature branch from the latest main commit. It creates a linear history with no merge commits.

Rebase vs Merge
BEFORE REBASE:
C1 → C2 → C3 ← main
↘ F1 → F2 ← feature
AFTER REBASE (feature rebased onto main):
C1 → C2 → C3 → F1' → F2' ← feature (new hashes!)
🔀 git rebase
# While on feature branch, rebase onto main
git switch feature/login
git rebase main

# If conflicts arise during rebase, resolve then:
git rebase --continue

# Abort the rebase entirely
git rebase --abort
🚫
Golden Rule of Rebasing

Never rebase commits that have been pushed to a shared remote branch. Rebasing rewrites commit hashes — if others have built on those commits, you'll cause major conflicts. Only rebase local, unpublished commits.

Cherry Pick — Apply Specific Commits

Cherry Picking

Cherry pick lets you apply a specific commit from any branch onto your current branch — without merging the whole branch. Useful for hotfixes or bringing a single feature across.

🍒 git cherry-pick
# Apply a single commit to current branch
git cherry-pick abc1234

# Apply multiple commits
git cherry-pick abc1234 def5678

# Apply a range of commits (exclusive of first, inclusive of last)
git cherry-pick abc1234..ghi9012

# Cherry pick but don't commit yet — just stage the changes
git cherry-pick -n abc1234

# Pick a specific file from another branch (not cherry-pick but similar)
git restore --source=feature/login -- src/LoginForm.tsx

Remote Repositories

What is a Remote?

A remote is a version of your repository stored on a server (like GitHub). origin is the conventional name for your primary remote. You can have multiple remotes (e.g., origin for your fork, upstream for the original repo).

🌐 Managing Remotes
# List all remotes
git remote -v
# origin  https://github.com/user/repo.git (fetch)
# origin  https://github.com/user/repo.git (push)

# Add a remote
git remote add origin https://github.com/user/repo.git

# Rename a remote
git remote rename origin backup

# Remove a remote
git remote rm backup

# Show detailed info about a remote
git remote show origin

Cloning a Repository

📥 git clone
# Clone into a folder named after the repo
git clone https://github.com/user/repo.git

# Clone into a custom folder name
git clone https://github.com/user/repo.git my-project

# Shallow clone — only last 1 commit (faster for large repos)
git clone --depth=1 https://github.com/user/repo.git

Push, Pull & Fetch

Fetching vs Pulling

fetch vs pull
GitHub (remote) ──git fetch──▶ origin/main (local copy)
then manually: git merge origin/main
GitHub (remote) ──git pull──▶ fetch + merge in one step
🔄 Fetch, Pull, Push
# Fetch — download remote changes WITHOUT merging
git fetch origin
git fetch               # fetches default remote

# Pull — fetch + merge (or rebase)
git pull
git pull origin main    # explicit remote + branch
git pull --rebase       # fetch + rebase instead of merge

# Push — upload your commits to remote
git push
git push origin main

# First push: set upstream tracking (-u)
git push -u origin main
# After this, 'git push' and 'git pull' work without arguments

# Push a local branch to remote (creates it there)
git push -u origin feature/login

# Delete a remote branch
git push origin --delete feature/login
Exercise 18 Collaborate with GitHub
  1. Create a GitHub repository and push your local git-practice repo to it
  2. On GitHub, edit a file directly in the browser — add a line
  3. Back locally, run git fetch then git log origin/main --oneline — see the remote change
  4. Run git pull to merge it locally
  5. Create a new local branch, make commits, push it, and delete the remote branch when done
💡 Hint: Use git log --oneline --graph --all to visualize all branches including remotes.

Pull Requests, Issues & Collaboration

The Collaboration Workflow

Feature Branch → Pull Request → Merge
1. Create branch → 2. Commit work → 3. Push branch → 4. Open PR
↓
5. Code Review → 6. Fix feedback → 7. Approve & Merge → 8. Delete branch

Issues — Tracking Work

  • Issues are GitHub's built-in task tracker — use them for bugs, features, and questions
  • Labels categorize issues (bug, enhancement, good first issue, help wanted)
  • Milestones group issues into a release goal (e.g., "v2.0 Release")
  • Assignees indicate who is responsible
  • Reference issues in commits: git commit -m "Fix #42 — null check on login" — GitHub auto-links and auto-closes
💡
Closing Issues via Commits

Include Closes #42, Fixes #15, or Resolves #7 in your commit message or PR description. GitHub automatically closes the linked issue when the PR is merged.

Releases

🚀 Creating a GitHub Release
# 1. Tag the commit you want to release
git tag -a v1.0.0 -m "v1.0.0 — Initial stable release"

# 2. Push the tag to GitHub
git push origin v1.0.0

# 3. On GitHub: Releases → Draft a new release → Select v1.0.0 tag
# 4. Write release notes, attach binaries if needed, Publish

Forks & Contributing to Open Source

Fork Workflow

When you don't have write access to a repo (e.g., open source projects), you fork it — creating your own copy on GitHub — then submit changes via Pull Request.

🍴 Fork → Contribute Workflow
# 1. Fork the repo on GitHub (click Fork button)

# 2. Clone YOUR fork
git clone https://github.com/YOUR_USER/repo.git

# 3. Add the original repo as 'upstream'
git remote add upstream https://github.com/ORIGINAL_OWNER/repo.git

# 4. Create a feature branch
git switch -c fix/typo-in-readme

# 5. Make changes, commit, push to YOUR fork
git push -u origin fix/typo-in-readme

# 6. On GitHub, open a Pull Request from your fork to the original

# KEEPING YOUR FORK IN SYNC with original:
git fetch upstream
git switch main
git merge upstream/main
git push origin main
21

Undoing & Amending Commits

Amend — Fix Your Last Commit

✏️ git commit --amend
# Fix the last commit message
git commit --amend -m "Correct commit message"

# Add a forgotten file to the last commit
git add forgotten-file.js
git commit --amend --no-edit
# --no-edit = keep the same commit message

# ⚠️ Amend rewrites history — only do this BEFORE pushing!

git reset — Move HEAD Backward

⏮️ git reset — three modes
# --soft: undo commit, keep changes STAGED
git reset --soft HEAD~1
# Use when: want to recommit with different message/content

# --mixed (default): undo commit, keep changes UNSTAGED
git reset HEAD~1
# Use when: want to review and re-stage selectively

# --hard: undo commit AND discard all changes (DESTRUCTIVE!)
git reset --hard HEAD~1
# Use when: commit was completely wrong, throw everything away
ModeCommit undone?Staging AreaWorking Directory
--soft✅ YesKept (staged)Unchanged
--mixed✅ YesClearedUnchanged
--hard✅ YesCleared⚠️ Discarded!

git revert — Safe Undo for Shared Branches

Unlike reset, git revert creates a new commit that undoes a past commit — it doesn't rewrite history, making it safe to use on shared/pushed branches.

🔙 git revert
# Create a new commit that reverses the changes of abc1234
git revert abc1234

# Revert without auto-committing (lets you edit the revert)
git revert -n abc1234

# Example: you pushed a bad commit → use revert, not reset
git revert HEAD    # undo the very last commit safely
git push           # safe to push — history still intact

Recovering Lost Commits — git reflog

🆘 git reflog — the safety net
# View every action HEAD has ever been at (like an undo history)
git reflog
# HEAD@{0}: commit: Add login feature
# HEAD@{1}: reset: moving to HEAD~2  ← you reset here
# HEAD@{2}: commit: Fix typo
# HEAD@{3}: commit: Initial commit

# Recover by resetting to the lost commit's hash
git reset --hard HEAD@{2}
# Or create a new branch from it
git branch recovered-work HEAD@{3}

Interactive Rebase — History Editing

What You Can Do with Interactive Rebase

Interactive rebase opens an editor with a list of commits. You can reorder, squash, edit, rename, or drop any of them before they land in history.

🔧 git rebase -i
# Open interactive rebase for last 4 commits
git rebase -i HEAD~4

# Editor shows (oldest first):
pick a1b2c3d Add login form
pick e4f5g6h Fix typo in login form
pick i7j8k9l Add signup form
pick m0n1o2p WIP: signup validation

# Available commands you can use per line:
# pick   = keep commit as-is
# reword = keep but edit commit message
# edit   = stop to amend the commit
# squash = meld into PREVIOUS commit (combine messages)
# fixup  = meld into previous commit (discard this message)
# drop   = delete this commit entirely
🗜️ Squash WIP commits into one clean commit
# Before: messy history
# pick a1b2c3d Add login form
# pick e4f5g6h Fix typo in login form   ← fixup these two
# pick i7j8k9l Add signup form
# pick m0n1o2p WIP: signup validation   ← fixup into signup

# Change to:
pick   a1b2c3d Add login form
fixup  e4f5g6h Fix typo in login form
pick   i7j8k9l Add signup form
fixup  m0n1o2p WIP: signup validation

# Result: 2 clean commits instead of 4 messy ones
⚠️
Only Rewrite Local History

Interactive rebase changes commit hashes. Never use it on commits already pushed to a shared remote branch — this will cause conflicts for all collaborators.

GitHub Connection Template

Scenario A — Push an Existing Local Repo to GitHub

You already have a local Git repo with commits. You've created a new empty GitHub repository and want to push your work there.

🚀 Push existing repo → GitHub
# Step 1: Connect your local repo to the GitHub remote
git remote add origin https://github.com/Akhilbille/sd.git

# Step 2: Rename default branch to 'master' (or 'main' — your choice)
git branch -M master

# Step 3: Push and set upstream tracking (-u means "track this remote")
git push -u origin master
# After this first push, future pushes just need: git push

Scenario B — Brand New Project from Scratch

You have no local files yet. You're starting completely fresh, connecting straight to a GitHub repo.

🆕 New project → GitHub from scratch
# Step 1: Create README and initialize Git
echo "# sd" >> README.md
git init

# Step 2: Stage and commit the first file
git add README.md
git commit -m "first commit"

# Step 3: Set branch name
git branch -M master

# Step 4: Connect to GitHub remote
git remote add origin https://github.com/Akhilbille/sd.git

# Step 5: Push and set upstream
git push -u origin master

Scenario C — Using main Instead of master

GitHub now defaults to main. If you see "branch 'master' set up to track 'origin/main'" errors, use this template instead.

🆕 New project with 'main' branch (modern)
# Initialize and set global default to 'main'
git config --global init.defaultBranch main

echo "# my-project" >> README.md
git init
git add README.md
git commit -m "Initial commit"

# Branch should already be 'main' now
git remote add origin https://github.com/USERNAME/REPO.git
git push -u origin main

Pro Tip: Set a Global .gitignore

Create a global gitignore so you never accidentally commit OS files: git config --global core.excludesfile ~/.gitignore_global, then add .DS_Store, Thumbs.db, and .env to that file.

⚠️
HTTPS vs SSH Authentication

GitHub deprecated password authentication. Use a Personal Access Token (PAT) when prompted for a password over HTTPS, or set up SSH keys for a smoother experience: ssh-keygen -t ed25519 -C "you@email.com", then add the public key to GitHub Settings → SSH Keys.
