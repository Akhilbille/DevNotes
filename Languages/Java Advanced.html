Programming Paradigms & OOP Benefits

What is a Programming Paradigm?

A paradigm is a style or approach to programming. Different paradigms give you different tools for structuring your code. Java primarily supports two major paradigms.

📜 Procedural Programming

Code is a sequence of instructions. Data and functions are separate. Good for simple scripts, but becomes hard to manage as programs grow.

🧩 Object-Oriented Programming

Code is organized around objects — entities that bundle data (fields) and behavior (methods) together. Scales well to large systems.

⚡ Functional Programming

Code is built from pure functions with no side effects. Java 8+ introduced streams and lambdas to support this style.

🎯 Event-Driven

Code runs in response to events (user clicks, messages). Common in GUI and server programming. Often layered on top of OOP.

The 4 Pillars of OOP

PillarMeaningJava Keyword / Tool
EncapsulationBundle data + methods; hide internalsprivate, getters/setters
AbstractionExpose only what's needed; hide complexityabstract, interfaces
InheritanceA class can reuse another class's behaviorextends
PolymorphismOne interface, many implementationsMethod overriding, interfaces

02

Classes & Objects

What is a Class?

A class is a blueprint or template. It defines what data (fields) and what actions (methods) every object of that type will have. An object is a concrete instance created from that blueprint.

💡
Class vs Object

Think of a class as a cookie cutter and objects as the cookies. The cutter defines the shape; you can make many cookies from one cutter, each independent.

📄 Person.java — Defining a Class 
public class Person {
    // Fields (instance variables) — each object has its own copy
    private String name;
    private int age;

    // Constructor — called when object is created with 'new'
    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // Method — defines behavior
    public void greet() {
        System.out.println("Hi, I am " + name + " and I am " + age + " years old.");
    }
}
📄 Main.java — Creating Objects 
public class Main {
    public static void main(String[] args) {
        // Create two independent Person objects
        Person alice = new Person("Alice", 30);
        Person bob   = new Person("Bob",   25);

        alice.greet(); // Hi, I am Alice and I am 30 years old.
        bob.greet();   // Hi, I am Bob and I am 25 years old.
    }
}

Memory Allocation

When you call new Person(...), Java allocates memory on the heap for the object. The variable (alice) holds a reference (pointer) to that memory, stored on the stack.

⚠️
Reference vs Value

Primitive types (int, boolean, etc.) are stored directly on the stack. Objects store a reference. Assigning Person b = a makes both variables point to the same object — not a copy!

03

Encapsulation — Getters & Setters

Why Encapsulate?

Without encapsulation, any code anywhere can change your object's data directly — making bugs hard to trace. Encapsulation means making fields private and exposing controlled access via getters and setters.

📄 Employee.java — Encapsulation in Practice 
public class Employee {
    private String name;
    private double salary;

    public Employee(String name, double salary) {
        this.name = name;
        setSalary(salary); // use setter for validation
    }

    // Getter — read-only access
    public String getName()   { return name; }
    public double getSalary() { return salary; }

    // Setter — controlled write with validation
    public void setSalary(double salary) {
        if (salary < 0)
            throw new IllegalArgumentException("Salary cannot be negative");
        this.salary = salary;
    }
}

// Usage:
Employee emp = new Employee("Alice", 50000);
emp.setSalary(-1000); // ❌ throws IllegalArgumentException
emp.setSalary(60000);  // ✅ valid
✅
Key Rule

Always make fields private. Only expose what callers need. Validation in setters protects your object's invariants — rules that must always hold true (e.g., salary ≥ 0).

04

Abstraction & Coupling

Abstraction

Abstraction means showing only the essential details and hiding the complexity. Like driving a car — you use the steering wheel and pedals without knowing how the engine works internally.

📄 MailService.java — Abstraction Example 
public class MailService {
    // Public method — the abstraction. Callers only see this.
    public void sendEmail(String to, String subject, String body) {
        connect();
        authenticate();
        transmit(to, subject, body);
        disconnect();
    }

    // Private helpers — hidden complexity
    private void connect()     { /* open socket */ }
    private void authenticate(){ /* login to SMTP server */ }
    private void transmit(String to, String subject, String body) { /* send */ }
    private void disconnect() { /* close socket */ }
}

Coupling

Coupling is how much one class depends on the internals of another. High coupling means changing one class forces changes in others — fragile code. Aim for low coupling.

❌ High Coupling 
// Order depends on SqlDatabase internals
class Order {
  SqlDatabase db = new SqlDatabase();
  void save() {
    db.openConnection();
    db.executeSql(...);
    db.closeConnection();
  }
}
✅ Low Coupling 
// Order depends only on an interface
class Order {
  Repository repo;
  Order(Repository r) { repo = r; }
  void save() {
    repo.save(this);
  }
}

Constructors & Method Overloading

Constructors

A constructor is a special method called when an object is created. It initializes the object's fields. Constructors have the same name as the class and no return type.

📄 Product.java — Constructor Overloading 
public class Product {
    private String name;
    private double price;
    private int stock;

    // Default constructor
    public Product() {
        this("Unknown", 0.0, 0); // calls the 3-param constructor
    }

    // 2-param constructor
    public Product(String name, double price) {
        this(name, price, 0); // calls the 3-param constructor
    }

    // Full constructor — do the real work here
    public Product(String name, double price, int stock) {
        this.name  = name;
        this.price = price;
        this.stock = stock;
    }
}

// All valid:
Product p1 = new Product();
Product p2 = new Product("Laptop", 999.99);
Product p3 = new Product("Phone", 699.0, 50);

Method Overloading

Overloading means having multiple methods with the same name but different parameter lists. Java picks the right one at compile time based on the arguments you pass.

📄 Calculator.java — Method Overloading 
public class Calculator {
    public int    add(int a, int b)          { return a + b; }
    public double add(double a, double b)  { return a + b; }
    public int    add(int a, int b, int c) { return a + b + c; }
    public String add(String a, String b)  { return a + b; }
}

Calculator calc = new Calculator();
calc.add(2, 3);             // → 5 (int version)
calc.add(2.5, 3.1);        // → 5.6 (double version)
calc.add("Hello, ", "World"); // → "Hello, World"

Static Members

Static Fields & Methods

A static member belongs to the class, not to any individual object. All instances share the same static field. Static methods can be called without creating an object.

📄 Counter.java — Static Field 
public class Counter {
    private static int count = 0; // shared across ALL instances
    private int id;

    public Counter() {
        count++;          // increment shared counter
        id = count;       // give this instance a unique id
    }

    public static int getCount() { return count; }
    public int getId() { return id; }
}

Counter c1 = new Counter(); // count = 1, c1.id = 1
Counter c2 = new Counter(); // count = 2, c2.id = 2
Counter c3 = new Counter(); // count = 3, c3.id = 3

System.out.println(Counter.getCount()); // 3
⚠️
Static Methods Cannot Access Instance Members

Inside a static method, this doesn't exist. You can only access other static members. Use static for utility methods (like Math.abs()) or factory methods.

📄 MathUtils.java — Static Utility Methods 
public class MathUtils {
    // No need to instantiate — all methods are static
    public static int    max(int a, int b) { return a > b ? a : b; }
    public static double circle(double r) { return Math.PI * r * r; }
    public static boolean isEven(int n)  { return n % 2 == 0; }
}

System.out.println(MathUtils.max(10, 20));    // 20
System.out.println(MathUtils.circle(5));    // 78.53...
System.out.println(MathUtils.isEven(7));   // false

Refactoring Towards OO Design

The Refactoring Process

Refactoring means restructuring existing code without changing its behavior. The goal is to improve readability, reduce coupling, and make the code easier to extend. The key steps are:

  • Identify responsibilities — what distinct jobs does your code perform?
  • Extract each responsibility into its own class
  • Move static members to the class that owns them
  • Replace duplicated logic with shared helper methods
  • Wire classes together via constructors (dependency injection)
📄 Before — God Class (everything in main) 
public class Main {
    static double rate = 0.05;

    public static void main(String[] args) {
        double principal = 100000;
        int years = 30;
        double payment = calcPayment(principal, rate, years);
        System.out.printf("Monthly: $%.2f%n", payment);
    }

    static double calcPayment(double p, double r, int y) {
        double n = y * 12;
        double mr = r / 12;
        return p * mr / (1 - Math.pow(1 + mr, -n));
    }
}
📄 After — Proper OO Design 
// MortgageCalculator.java
public class MortgageCalculator {
    private final double principal;
    private final double annualRate;
    private final int years;

    public MortgageCalculator(double p, double r, int y) {
        principal = p; annualRate = r; years = y;
    }

    public double calculateMonthlyPayment() {
        double n  = years * 12;
        double mr = annualRate / 12;
        return principal * mr / (1 - Math.pow(1 + mr, -n));
    }
}

// MortgageReport.java
public class MortgageReport {
    private final MortgageCalculator calculator;

    public MortgageReport(MortgageCalculator calc) {
        this.calculator = calc;
    }

    public void print() {
        System.out.printf("Monthly Payment: $%.2f%n",
            calculator.calculateMonthlyPayment());
    }
}

// Main.java
MortgageCalculator calc = new MortgageCalculator(100000, 0.05, 30);
new MortgageReport(calc).print();

Inheritance

What is Inheritance?

Inheritance allows a class (child/subclass) to inherit fields and methods from another class (parent/superclass) using the extends keyword. This promotes code reuse.

📄 Animal hierarchy 
// Parent class
public class Animal {
    private String name;

    public Animal(String name) { this.name = name; }

    public String getName() { return name; }

    public void eat() {
        System.out.println(name + " is eating.");
    }
}

// Child class inherits from Animal
public class Dog extends Animal {
    private String breed;

    public Dog(String name, String breed) {
        super(name); // call parent constructor
        this.breed = breed;
    }

    public void bark() {
        System.out.println(getName() + " says: Woof!");
    }
}

Dog dog = new Dog("Rex", "Labrador");
dog.eat();  // inherited: Rex is eating.
dog.bark(); // own: Rex says: Woof!

The Object Class

Every class in Java implicitly extends Object. This gives all objects methods like toString(), equals(), and hashCode(). You can override these to customize behavior.

📄 Overriding toString() and equals() 
public class Point {
    private int x, y;

    public Point(int x, int y) { this.x = x; this.y = y; }

    @Override
    public String toString() {
        return "(" + x + ", " + y + ")";
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (!(obj instanceof Point)) return false;
        Point other = (Point) obj;
        return this.x == other.x && this.y == other.y;
    }
}

Point p1 = new Point(3, 4);
Point p2 = new Point(3, 4);
System.out.println(p1);            // (3, 4)
System.out.println(p1.equals(p2)); // true

Upcasting & Downcasting

Upcasting (treating a Dog as an Animal) is always safe and automatic. Downcasting (treating an Animal as a Dog) requires a cast and can throw ClassCastException — use instanceof first.

📄 Casting 
Animal a = new Dog("Rex", "Lab"); // upcast — safe, automatic
a.eat();                            // ✅ works

// Downcast — must check first
if (a instanceof Dog dog) {         // Java 16+ pattern matching
    dog.bark();                      // ✅ safe
}

Polymorphism

What is Polymorphism?

Polymorphism means "many forms." In Java, the same method call can do different things depending on the actual object type at runtime. This enables writing code that works with any subclass — past, present, or future.

📄 Runtime Polymorphism 
public class UIControl {
    public void draw() { System.out.println("Drawing a control"); }
}

public class Button extends UIControl {
    @Override
    public void draw() { System.out.println("Drawing a Button"); }
}

public class TextBox extends UIControl {
    @Override
    public void draw() { System.out.println("Drawing a TextBox"); }
}

// Polymorphic code — works for any UIControl subclass
public static void drawAll(UIControl[] controls) {
    for (UIControl c : controls)
        c.draw(); // calls correct version at RUNTIME
}

UIControl[] controls = {
    new Button(),
    new TextBox(),
    new Button()
};
drawAll(controls);
// Drawing a Button
// Drawing a TextBox
// Drawing a Button
✅
Why Polymorphism is Powerful

You can add a new CheckBox extends UIControl tomorrow and pass it to drawAll — no changes to existing code. This is the Open/Closed Principle: open for extension, closed for modification.

Abstract Classes & Final

Abstract Classes

An abstract class cannot be instantiated — it's a template. It can have both concrete (implemented) and abstract (unimplemented) methods. Subclasses must implement all abstract methods.

📄 Abstract Class Example 
public abstract class Shape {
    private String color;

    public Shape(String color) { this.color = color; }

    // Abstract — subclass MUST implement
    public abstract double area();

    // Concrete — subclass inherits this for free
    public void printInfo() {
        System.out.printf("%s %s — area: %.2f%n", color, getClass().getSimpleName(), area());
    }
}

public class Circle extends Shape {
    private double radius;
    public Circle(String color, double r) { super(color); radius = r; }

    @Override
    public double area() { return Math.PI * radius * radius; }
}

// Shape s = new Shape(...); ❌ Cannot instantiate abstract class!
Shape c = new Circle("Red", 5);
c.printInfo(); // Red Circle — area: 78.54

Final Classes & Methods

The final keyword prevents inheritance or overriding. A final class cannot be extended. A final method cannot be overridden. A final variable is a constant.

📄 Final Usage 
public final class String { /* immutable — cannot be extended */ }

public class Base {
    public final void criticalMethod() {
        // cannot be overridden in subclasses
    }
}

public class Constants {
    public static final double PI = 3.14159; // compile-time constant
    public static final int MAX_USERS = 1000;
}

Interfaces

What is an Interface?

An interface is a contract — a list of method signatures that implementing classes must provide. Unlike abstract classes, interfaces define what a class can do, not what it is. A class can implement multiple interfaces.

📄 Printable.java & Resizable.java 
public interface Printable {
    void print(); // implicitly public abstract
}

public interface Resizable {
    void resize(double factor);
}

// Implementing MULTIPLE interfaces
public class Image implements Printable, Resizable {
    private String filename;
    private double width, height;

    public Image(String f, double w, double h) {
        filename = f; width = w; height = h;
    }

    @Override
    public void print() {
        System.out.println("Printing " + filename + " [" + width + "x" + height + "]");
    }

    @Override
    public void resize(double factor) {
        width *= factor; height *= factor;
    }
}

Default & Static Interface Methods (Java 8+)

📄 Interface with default and static methods 
public interface Logger {
    void log(String message);

    // Default — all implementors get this for free
    default void logError(String msg) {
        log("[ERROR] " + msg);
    }

    // Static — called on the interface itself
    static Logger console() {
        return msg -> System.out.println(msg);
    }
}

Logger log = Logger.console(); // static factory
log.log("App started");
log.logError("Something broke"); // default method

Dependency Injection

What is DI?

Dependency Injection means providing a class's dependencies from outside, rather than the class creating them itself. This decouples classes and makes them testable. There are 3 types:

📄 Three Types of DI 
public interface TaxCalculator {
    double calculateTax(double income);
}

public class TaxReport {
    private TaxCalculator calculator;

    // 1. Constructor Injection — preferred
    public TaxReport(TaxCalculator calc) {
        this.calculator = calc;
    }

    // 2. Setter Injection — use if dependency is optional
    public void setCalculator(TaxCalculator calc) {
        this.calculator = calc;
    }

    // 3. Method Injection — pass dependency to individual method
    public void show(TaxCalculator calc, double income) {
        System.out.println("Tax: $" + calc.calculateTax(income));
    }

    public void show(double income) {
        System.out.println("Tax: $" + calculator.calculateTax(income));
    }
}

// Wire it up:
TaxCalculator flat = income -> income * 0.3;
TaxReport report = new TaxReport(flat);
report.show(75000); // Tax: $22500.0

// Swap the calculator — no change to TaxReport!
report.setCalculator(income -> income * 0.2);
report.show(75000); // Tax: $15000.0

Interface Segregation Principle

The Problem with Fat Interfaces

The Interface Segregation Principle (ISP) says: don't force a class to implement methods it doesn't need. Split large ("fat") interfaces into smaller, focused ones.

❌ Fat Interface 
interface UIWidget {
  void drag();
  void resize();
  void render();
  void click();
}
// Checkbox must implement drag() even if it
// doesn't support dragging — bad!
✅ Segregated Interfaces 
interface Draggable { void drag(); }
interface Resizable  { void resize(); }
interface Renderable { void render(); }
interface Clickable  { void click(); }
// Checkbox implements only what it needs:
// Renderable + Clickable

Exceptions

Exception Hierarchy

In Java, all exceptions inherit from Throwable. The two main branches are checked (must be handled) and unchecked (runtime, optional to handle).

Hierarchy
Throwable → Error (JVM crashes: OutOfMemoryError)
Throwable → Exception → checked (IOException, SQLException)
Throwable → Exception → RuntimeException → unchecked (NullPointerException, IllegalArgumentException)
📄 Catching Exceptions 
import java.io.*;
import java.util.*;

public class FileReader {
    public static void read(String path) {
        // try-with-resources — auto closes the reader
        try (Scanner sc = new Scanner(new File(path))) {
            while (sc.hasNextLine())
                System.out.println(sc.nextLine());
        } catch (FileNotFoundException e) {
            System.err.println("File not found: " + path);
        } catch (IOException e) {
            System.err.println("IO error: " + e.getMessage());
        } finally {
            System.out.println("Done reading."); // always runs
        }
    }
}

Custom Exceptions

📄 Custom Exception + Chaining 
// Custom checked exception
public class InsufficientFundsException extends Exception {
    public InsufficientFundsException(double amount) {
        super("Insufficient funds. Needed: $" + amount);
    }
}

public class BankAccount {
    private double balance = 100;

    public void withdraw(double amount) throws InsufficientFundsException {
        if (amount > balance)
            throw new InsufficientFundsException(amount);
        balance -= amount;
    }
}

// Usage with exception chaining:
try {
    account.withdraw(500);
} catch (InsufficientFundsException e) {
    throw new RuntimeException("Transaction failed", e); // chain it
}

Generics

Why Generics?

Generics let you write classes and methods that work with any type while maintaining type safety at compile time. Without generics you'd need casting everywhere and risk ClassCastException at runtime.

📄 Generic Class 
// Generic stack — works with any type
public class Stack<T> {
    private ArrayList<T> items = new ArrayList<>();

    public void push(T item) { items.add(item); }

    public T pop() {
        if (items.isEmpty())
            throw new EmptyStackException();
        return items.remove(items.size() - 1);
    }

    public boolean isEmpty() { return items.isEmpty(); }
    public int     size()    { return items.size(); }
}

Stack<Integer> intStack = new Stack<>();
intStack.push(10);
intStack.push(20);
int top = intStack.pop(); // 20 — no cast needed!

Stack<String> strStack = new Stack<>();
strStack.push("hello");
String s = strStack.pop(); // "hello" — type safe

Constraints (Bounded Type Parameters)

📄 Upper Bounded & Generic Methods 
// T must implement Comparable
public class Sorter<T extends Comparable<T>> {
    public T max(List<T> list) {
        T result = list.get(0);
        for (T item : list)
            if (item.compareTo(result) > 0)
                result = item;
        return result;
    }
}

// Generic static method
public static <T> void swap(T[] arr, int i, int j) {
    T temp = arr[i];
    arr[i] = arr[j];
    arr[j] = temp;
}

// Wildcard — accept List of any subtype of Number
public static double sum(List<? extends Number> list) {
    double total = 0;
    for (Number n : list) total += n.doubleValue();
    return total;
}

📄 List, Set, Map in Action 
// List — ordered, duplicates allowed
List<String> names = new ArrayList<>(List.of("Alice", "Bob", "Alice"));
names.sort(Comparator.naturalOrder());
System.out.println(names); // [Alice, Alice, Bob]

// Set — no duplicates
Set<String> unique = new HashSet<>(names);
System.out.println(unique); // [Alice, Bob]

// Map — key → value
Map<String, Integer> scores = new HashMap<>();
scores.put("Alice", 95);
scores.put("Bob", 82);
scores.put("Alice", 98); // overwrites 95
scores.getOrDefault("Charlie", 0); // → 0 (safe)

for (var entry : scores.entrySet())
    System.out.println(entry.getKey() + ": " + entry.getValue());

// Queue (FIFO)
Queue<String> queue = new ArrayDeque<>();
queue.offer("task1");
queue.offer("task2");
System.out.println(queue.poll()); // task1

Sorting with Comparable & Comparator

📄 Custom Sorting 
public class Student implements Comparable<Student> {
    public String name;
    public double gpa;

    @Override
    public int compareTo(Student other) {
        return Double.compare(this.gpa, other.gpa); // natural order: ascending GPA
    }
}

List<Student> students = new ArrayList<>();
// Sort by name using Comparator (doesn't change natural order)
students.sort(Comparator.comparing(s -> s.name));
// Sort by GPA descending then name ascending
students.sort(Comparator.comparingDouble((Student s) -> s.gpa)
    .reversed()
    .thenComparing(s -> s.name));

Lambda Expressions & Functional Interfaces

What is a Lambda?

A lambda expression is a concise way to implement a functional interface (an interface with exactly one abstract method). Instead of an anonymous class, you write an arrow function.

📄 Lambda Syntax Evolution 
// Old way — anonymous inner class
Comparator<String> cmp = new Comparator<String>() {
    public int compare(String a, String b) {
        return a.compareTo(b);
    }
};

// Lambda — much cleaner
Comparator<String> cmpLambda = (a, b) -> a.compareTo(b);

// Method reference — even shorter
Comparator<String> cmpRef = String::compareTo;

Built-in Functional Interfaces

📄 Consumer, Supplier, Function, Predicate 
import java.util.function.*;

// Consumer<T> — takes T, returns nothing
Consumer<String> print = msg -> System.out.println(msg);
Consumer<String> shout = msg -> System.out.println(msg.toUpperCase());
print.andThen(shout).accept("hello"); // hello \n HELLO

// Supplier<T> — takes nothing, returns T
Supplier<String> greeting = () -> "Good morning!";
System.out.println(greeting.get()); // Good morning!

// Function<T, R> — takes T, returns R
Function<String, Integer> length = String::length;
Function<Integer, Boolean> isLong = n -> n > 5;
Function<String, Boolean> isLongWord = length.andThen(isLong);
isLongWord.apply("hello");   // false (5 chars)
isLongWord.apply("programming"); // true

// Predicate<T> — takes T, returns boolean
Predicate<Integer> isEven = n -> n % 2 == 0;
Predicate<Integer> isPos  = n -> n > 0;
Predicate<Integer> isEvenAndPos = isEven.and(isPos);
isEvenAndPos.test(4);  // true
isEvenAndPos.test(-2); // false

// BinaryOperator<T> — takes two T, returns T
BinaryOperator<Integer> add = (a, b) -> a + b;
System.out.println(add.apply(10, 20)); // 30

Streams

What are Streams?

A Stream is a sequence of elements that you can process in a functional, declarative style. Streams don't store data — they process it lazily. A stream pipeline has: source → intermediate operations → terminal operation.

📄 Stream Pipeline Examples 
import java.util.stream.*;
import java.util.*;

List<Integer> numbers = List.of(5, 3, 8, 1, 9, 2, 7, 4, 6);

// Filter → Map → Sorted → Collect
List<String> result = numbers.stream()
    .filter(n -> n % 2 == 0)       // keep even: 8,2,4,6
    .sorted()                       // 2,4,6,8
    .map(n -> "#" + n)              // "#2","#4","#6","#8"
    .collect(Collectors.toList()); // collect to List

// Reduce
int sum = numbers.stream()
    .reduce(0, Integer::sum); // 45

// Simple reducers
int max = numbers.stream().max(Integer::compare).get(); // 9
long count = numbers.stream().filter(n -> n > 5).count(); // 4

// Grouping
Map<Boolean, List<Integer>> partitioned = numbers.stream()
    .collect(Collectors.partitioningBy(n -> n % 2 == 0));
// {false=[5,3,1,9,7], true=[8,2,4,6]}

List<String> words = List.of("apple", "banana", "avocado", "blueberry");
Map<Character, List<String>> grouped = words.stream()
    .collect(Collectors.groupingBy(w -> w.charAt(0)));
// {a=[apple, avocado], b=[banana, blueberry]}

Creating Streams

📄 Stream Sources 
Stream.of(1, 2, 3);              // from values
list.stream();                    // from collection
Arrays.stream(arr);               // from array
IntStream.range(0, 10);          // 0..9 (primitive stream)
IntStream.rangeClosed(1, 10);    // 1..10
Stream.generate(() -> "hello");  // infinite — use limit()
Stream.iterate(1, n -> n * 2);   // infinite: 1,2,4,8... — use limit()

// Slicing
numbers.stream().limit(3);       // first 3
numbers.stream().skip(2);        // skip first 2
numbers.stream().distinct();     // unique only

Concurrency & Multi-threading

Processes & Threads

A process is a running program with its own memory space. A thread is a lightweight unit of execution within a process — threads share memory. Java apps start with one thread: the main thread.

📄 Creating & Running Threads 
// Method 1: Extend Thread
class DownloadThread extends Thread {
    private String file;
    DownloadThread(String f) { file = f; }

    @Override
    public void run() {
        System.out.println("Downloading: " + file);
    }
}

// Method 2: Implement Runnable (preferred — allows extending other classes)
Thread t1 = new Thread(() -> System.out.println("Task 1 running"));
Thread t2 = new Thread(() -> System.out.println("Task 2 running"));

t1.start(); // start — runs run() on a new thread
t2.start();
t1.join();  // wait for t1 to finish before continuing

System.out.println("All threads done");

Race Conditions & Thread Safety

A race condition occurs when two threads access shared data simultaneously and the result depends on timing. Solutions: synchronization, atomic variables, concurrent collections.

📄 Race Condition & Fix 
// ❌ Race condition — counter may not reach 10,000
class Counter {
    private int count = 0;
    public void increment() { count++; } // NOT atomic!
}

// ✅ Fix 1: synchronized method
public synchronized void increment() { count++; }

// ✅ Fix 2: AtomicInteger (fastest for single operations)
import java.util.concurrent.atomic.*;
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet();
counter.addAndGet(5);

// ✅ Fix 3: LongAdder (best for high-contention counters)
import java.util.concurrent.atomic.*;
LongAdder adder = new LongAdder();
adder.increment();
long total = adder.sum();

// ✅ Concurrent collections (thread-safe)
import java.util.concurrent.*;
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();

Optionals

The Null Problem

null is the most common source of bugs in Java — NullPointerException is infamous. Optional<T> is a container that either holds a value or is empty, making the possibility of "no value" explicit in the type system.

📄 Creating and Using Optionals 
import java.util.*;

// Creating Optionals
Optional<String> full  = Optional.of("hello");      // throws NPE if null
Optional<String> safe  = Optional.ofNullable(mayBeNull); // safe
Optional<String> empty = Optional.empty();          // explicitly empty

// Checking and extracting
if (full.isPresent()) System.out.println(full.get());
full.ifPresent(v -> System.out.println(v));   // cleaner

// Defaults
String val1 = empty.orElse("default");          // "default"
String val2 = empty.orElseGet(() -> computeDefault()); // lazy
String val3 = empty.orElseThrow(                  // throw if empty
    () -> new NoSuchElementException("Value missing"));
📄 Chaining Optionals (map, flatMap, filter) 
// Real-world example: user → address → city
public Optional<String> getCity(User user) {
    return Optional.ofNullable(user)
        .map(User::getAddress)         // Optional<Address>
        .map(Address::getCity)          // Optional<String>
        .filter(city -> !city.isEmpty()) // discard empty string
        .map(String::toUpperCase);      // Optional<String>
}

// vs the old way (null-check hell):
String city = null;
if (user != null && user.getAddress() != null &&
    user.getAddress().getCity() != null &&
    !user.getAddress().getCity().isEmpty()) {
    city = user.getAddress().getCity().toUpperCase();
}
🚫
Optional Anti-patterns

Never use Optional as a method parameter or field. Don't call get() without checking isPresent() first. Don't use Optional to replace every null check — only use it as a return type when a value may genuinely be absent.

📄 Optional with Streams 
List<String> names = List.of("Alice", "Bob", "Charlie");

// findFirst / findAny return Optional
Optional<String> first = names.stream()
    .filter(n -> n.startsWith("C"))
    .findFirst(); // Optional[Charlie]

first.ifPresent(System.out::println); // Charlie

// min / max return Optional
Optional<String> shortest = names.stream()
    .min(Comparator.comparingInt(String::length)); // Optional[Bob]

// Stream of Optionals → Stream of values (Java 9+)
List<Optional<String>> optionals = List.of(
    Optional.of("hello"),
    Optional.empty(),
    Optional.of("world")
);

List<String> present = optionals.stream()
    .flatMap(Optional::stream) // unwrap present values
    .collect(Collectors.toList()); // ["hello", "world"]
