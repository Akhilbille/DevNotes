Spring Framework Overview

What is Spring?

Spring is the most popular framework for building Java applications. It is modular — you only pick the modules you need. Spring handles boilerplate infrastructure so you can focus on business logic.

🌐 WEB Module

For building web applications and RESTful APIs. Provides Spring MVC, controllers, request mappings.

🗄️ DATA Module

For working with databases. Integrates JPA, Hibernate, Spring Data JPA repositories.

⚙️ CORE Module

The heart of Spring — handles Dependency Injection (DI) and manages all objects (Beans) in the IOC Container.

🔒 AOP Module

Aspect-Oriented Programming. Handles cross-cutting concerns like logging and security without cluttering main code.

🧪 TEST Module

For testing Spring components in isolation. Works with JUnit and Mockito.

🔐 SECURITY Module

Authentication & Authorization for web apps. Supports OAuth2, JWT, session management.

📐 Spring Ecosystem Diagram

The Problem Spring Solves

Without Spring Boot, setting up a Java web app requires manual effort:

  • Set up a web server (Tomcat, Jetty)
  • Configure routing manually
  • Manage all dependencies yourself
  • Write boilerplate code for every component
✅ Spring Boot to the Rescue Spring Boot is built on top of Spring Framework and takes care of all tedious setup automatically — embedded server, auto-configuration, sensible defaults. You focus purely on business logic.

Spring MVC Architecture

Spring uses the MVC (Model-View-Controller) architectural pattern to separate concerns cleanly.

📐 Spring MVC Flow
🗃️ Model

Contains business logic and connects to the database. Stores and processes data.

🖥️ View

What the user sees — HTML, CSS, JS. In modern Spring apps this is often a React/Angular SPA or Thymeleaf template.

🎮 Controller

Handles requests from users, interacts with the Model, and tells the View what to display. Acts as a mediator.

📍 @Value

Injects properties from application.properties at runtime: @Value("${app.page-size}")

📄 HomeController.java 
@Controller
public class HomeController {

    @Value("${spring.application.name}")
    private String appName; // Spring injects the value at runtime

    @RequestMapping("/")
    public String index() {
        System.out.println("appName: " + appName);
        return "index.html"; // serves resources/static/index.html
    }
}
💡
Static Files Location Static files (HTML, CSS, JS) must be placed in src/main/resources/static/. Spring Boot automatically serves files from this folder.

Dependency Injection, IOC & Beans

Core Concepts

🔄 IOC — Inversion of Control

Inverting (giving) control of object creation to the Spring Framework instead of creating objects manually with new.

💉 DI — Dependency Injection

Injecting a dependent object into a target class. Spring creates and wires the dependency for you.

🫘 Bean

An object that is created and managed by the Spring IOC Container. All Spring-managed objects are beans.

📦 IOC Container

Part of Spring CORE. Creates beans, manages their lifecycle, and injects dependencies automatically.

📐 Dependency Injection — E-Commerce Example
💡
IOC Container Initialization When you call SpringApplication.run(StoreApplication.class, args) it returns an ApplicationContext — the IOC container with all beans already created and wired.

Constructor Injection

The preferred way to inject mandatory dependencies. If there's only one constructor, @Autowired is optional. With multiple constructors it is required.

📄 OrderService.java — Constructor Injection 
@Service
public class OrderService {

    private PaymentService paymentService;

    // Constructor Injection — @Autowired optional for single constructor
    @Autowired
    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }

    public void placeOrder() {
        paymentService.processPayment(100);
    }
}

Setter Injection

Used for optional dependencies. The bean is created first, then the setter is called by Spring to inject the dependency.

📄 OrderService.java — Setter Injection 
@Service
public class OrderService {

    private PaymentService paymentService;

    @Autowired
    public void setPaymentService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
TypeUse WhenAnnotation
Constructor InjectionDependency is mandatory@Autowired (optional if 1 constructor)
Setter InjectionDependency is optional@Autowired on setter method

Stereotype Annotations

Spring uses Stereotype Annotations to identify classes it should manage as beans. Place these at class level.

@Component

Generic annotation. Marks any class as a Spring-managed bean. Use for utility classes.

@Service

Marks a class as a business logic service. Semantically clearer than @Component.

@Repository

Marks a class for database interaction. Also enables exception translation.

@Controller

Marks a class for handling web requests. Returns views or responses.

📄 OrderService.java — @Service Bean 
@Service   // Tells Spring to create and manage this bean
public class OrderService {
    // Spring will create ONE instance and manage it
}

Controlling Bean Selection — Multiple Implementations

When multiple beans qualify for the same dependency, Spring throws an exception. Use @Qualifier or @Primary to resolve the conflict.

📄 StripePaymentService.java + PaypalPaymentService.java 
@Service("stripe")
public class StripePaymentService implements PaymentService { ... }

@Service("paypal")
@Primary  // Default bean when no @Qualifier specified
public class PaypalPaymentService implements PaymentService { ... }
📄 OrderService.java — Using @Qualifier 
@Service
public class OrderService {
    private PaymentService paymentService;

    @Autowired
    public OrderService(@Qualifier("stripe") PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
Priority Rule If both @Primary and @Qualifier are used, @Qualifier always wins. It has higher priority.

Bean Configuration & Scopes

Externalizing Configuration

Never hardcode sensitive values like API keys in source code. Store them in application.properties or application.yaml and inject them with @Value.

📄 application.yaml — Stripe Config 
stripe:
  apiUrl: https://api.stripe.com
  timeout: 1000
  enabled: true
  supported-currencies: USD, EUR, GBP
📄 StripePaymentService.java — @Value Injection 
@Service("stripe")
public class StripePaymentService implements PaymentService {

    @Value("${stripe.apiUrl}")
    private String apiUrl;

    @Value("${stripe.supported-currencies}")
    private List<String> currencies;

    @Value("${stripe.timeout:3000}")  // 3000 is the DEFAULT if not found
    private int timeout;

    @Value("${stripe.enabled}")
    private boolean enabled;
}

Configuring Beans Using Java Code (@Configuration)

For third-party classes you can't annotate, use @Configuration + @Bean to create beans manually. This gives you full control over object creation.

📄 AppConfig.java — Java-Based Bean Config 
@Configuration
public class AppConfig {

    @Bean
    public PaymentService stripe() {
        return new StripePaymentService(); // manually create, Spring manages
    }

    @Bean
    public OrderService orderService() {
        return new OrderService(stripe()); // wire the dependency manually
    }
}
When to use @Configuration Use it when: (1) configuring third-party library objects, (2) creating beans conditionally, or (3) needing fine-grained control over initialization order.

Lazy Initialization

By default, Spring uses Eager Loading — it creates all beans at startup. With @Lazy, Spring delays bean creation until the bean is first requested.

📄 HeavyService.java — @Lazy Bean 
@Service
@Lazy  // Won't create until first requested by another bean
public class HeavyLoadService {

    public HeavyLoadService() {
        System.out.println("HeavyLoadService created!");
    }
}
Loading TypeWhen createdUse case
Eager (Default)At application startupMost beans — fail fast on config errors
Lazy (@Lazy)When first requestedHeavy, rarely-used services

Bean Lifecycle Hooks & YAML

Bean Lifecycle

Every Spring bean goes through a defined lifecycle. You can hook into this lifecycle using two key annotations.

📐 Bean Lifecycle Diagram
📄 OrderService.java — Lifecycle Hooks 
@Service
public class OrderService {

    // Constructor runs FIRST
    public OrderService() { }

    @PostConstruct
    public void init() {
        // Runs AFTER constructor + @Autowired wiring
        System.out.println("Initializing after creating constructor");
    }

    @PreDestroy
    public void cleanup() {
        // Runs BEFORE bean is destroyed (app shutdown)
        System.out.println("Destroying / cleaning...");
    }
}

JDBC, JPA & Hibernate

Data Access Layers Compared

📐 Data Access Layer Stack
🔌 JDBC

Java Database Connectivity. Allows Java to execute raw SQL. You manage connections, statements, and result sets manually — lots of boilerplate code.

📋 JPA

Jakarta Persistence API. A specification for mapping Java objects to database tables. You work with Java objects, not SQL. JPA is just a spec — needs an implementation.

🐻 Hibernate

ORM Framework. The most popular JPA implementation. Provides caching, HQL (Hibernate Query Language), and automatic schema generation.

🌱 Spring Data JPA

Built on top of JPA/Hibernate. Provides ready-made Repository interfaces so you write zero SQL for common operations.

Database Migration with Flyway

Flyway is a database migration plugin that runs SQL scripts automatically when your app starts. This ensures your database schema is always in the correct state.

📐 Flyway Migration Commands
CommandDescription
flyway:migrateRuns all pending SQL migration scripts to DB
flyway:cleanDeletes all tables from DB (disabled by default)
flyway:repairRemoves deleted migration files from DB tracking
flyway:infoShows all migration file info and status
flyway:validateValidates migration files — no DB changes
📄 V1__initial_migration.sql — Create Users Table 
-- File must follow: VN__description.sql (double underscore)
-- Located at: src/main/resources/db/migration/

CREATE TABLE users (
    id       BIGINT AUTO_INCREMENT PRIMARY KEY,
    name     VARCHAR(255) NOT NULL,
    email    VARCHAR(255) NOT NULL,
    password VARCHAR(255) NOT NULL
);

CREATE TABLE addresses (
    id      BIGINT AUTO_INCREMENT PRIMARY KEY,
    street  VARCHAR(255) NOT NULL,
    city    VARCHAR(255) NOT NULL,
    zip     VARCHAR(255) NOT NULL,
    user_id BIGINT NOT NULL,
    CONSTRAINT addresses_users_id_fk
        FOREIGN KEY (user_id) REFERENCES users (id)
);
Never Delete Migration Files If you made a mistake, create a new migration (V2, V3...) to fix it. Flyway tracks which scripts have run. Deleting them breaks the tracking.

Entities, Lombok & Relationships

Defining Entities

An Entity is a Java class that maps directly to a database table. Annotate it with @Entity and @Table.

📐 Application Model — Entity Relationships
📄 User.java — Entity Class 
@Entity
@Table(name = "users")
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, name = "name")
    private String name;

    @Column(nullable = false, name = "email")
    private String email;

    @Column(nullable = false, name = "password")
    private String password;

    // Getters & Setters (or use Lombok below)
}

Simplifying Code with Lombok

Lombok is a library that auto-generates boilerplate code (getters, setters, constructors, toString) using annotations — at compile time.

📄 User.java — With Lombok 
@Getter
@Setter
@AllArgsConstructor
@NoArgsConstructor
@ToString
@Builder
@Entity
@Table(name = "users")
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String email;
    private String password;
}

// Builder pattern — create objects fluently
var user = User.builder()
    .name("Akhil")
    .email("a@gmail.com")
    .password("@$123@")
    .build();
JPA Needs @NoArgsConstructor Always add both @AllArgsConstructor and @NoArgsConstructor when using JPA. JPA spec requires a no-arg constructor to create entity instances.
💡
@Builder.Default When using @Builder with fields that have default values (like empty lists), add @Builder.Default to prevent NullPointerExceptions: @Builder.Default private List<Address> addresses = new ArrayList<>();

One-to-Many Relationships

One User can have many Addresses, but each Address belongs to only one User. The FK lives on the "many" side (Address table stores user_id).

📐 One-to-Many — Database Table View
📐 Unidirectional vs Bidirectional
📄 User.java — @OneToMany (Bidirectional) 
@OneToMany(mappedBy = "user", cascade = {CascadeType.PERSIST, CascadeType.REMOVE},
           orphanRemoval = true)
@Builder.Default
private List<Address> addresses = new ArrayList<>();

// Helper method — always wire both sides together
public void addAddress(Address address) {
    addresses.add(address);
    address.setUser(this);
}

public void removeAddress(Address address) {
    addresses.remove(address);
    address.setUser(null);
}
📄 Address.java — @ManyToOne (Owner Side) 
@ManyToOne
@JoinColumn(name = "user_id")  // Address is OWNER (has the FK column)
@ToString.Exclude              // Prevent infinite loop in toString()
private User user;
💡
Who is the Owner? The entity that has the foreign key column in the database is the owner. Address has user_id FK, so Address is the owner. The owner side uses @JoinColumn. The non-owner side uses mappedBy.

Many-to-Many Relationships

A User can have many Tags, and a Tag can belong to many Users. This creates a join table (user_tags) in the database with two FK columns.

📐 Many-to-Many — Join Table in Database
📄 User.java — @ManyToMany (Owner) 
@ManyToMany
@JoinTable(
    name = "user_tags",
    joinColumns = @JoinColumn(name = "user_id"),
    inverseJoinColumns = @JoinColumn(name = "tag_id")
)
@Builder.Default
private Set<Tag> tags = new HashSet<>();

// Add tag helper
public void addTag(String tagName) {
    var tag = new Tag(tagName);
    tags.add(tag);
    tag.getUsers().add(this);
}

public void removeTag(Tag tag) {
    tags.remove(tag);
    tag.setUsers(null);
}
📄 Tag.java — Non-owner side 
@ManyToMany(mappedBy = "tags")
@ToString.Exclude  // Avoid StackOverflowException from circular toString
@Builder.Default
private Set<User> users = new HashSet<>();

One-to-One Relationships

📐 One-to-One — Database Table View
📄 User.java + Profile.java — @OneToOne 
// In User.java
@OneToOne(mappedBy = "user")
@MapsId
private Profile profile;

// In Profile.java — Profile is the OWNER (has user_id FK)
@OneToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "id")
@MapsId
@ToString.Exclude
private User user;

Fetching Strategies

📐 Eager vs Lazy Loading
StrategyDescriptionDefault for
FetchType.EAGERRelated objects loaded immediately with parent@OneToOne, @ManyToOne
FetchType.LAZYRelated objects loaded only when accessed@OneToMany, @ManyToMany
Best Practice Use Eager Loading when you always need related data. Use Lazy Loading when related data is rarely accessed. Lazy loading avoids unnecessary database queries.

Repositories & Transactions

Repository Hierarchy

Spring Data JPA provides ready-made repository interfaces. Extend one to get CRUD operations for free — no SQL needed.

📐 Repository Interface Hierarchy
Which to extend? Use CrudRepository for basic CRUD. Use JpaRepository for advanced features like batch operations, example-based queries, and custom flush behavior.
📄 UserRepository.java — Defining a Repository 
// Step 1: Create interface in a 'repositories' package
// Step 2: Extend CrudRepository<EntityClass, PrimaryKeyType>
public interface UserRepository extends CrudRepository<User, Long> {
    // Spring creates proxy implementation at runtime — no code needed!
}
📄 Using Repository — CRUD Operations 
var repository = context.getBean(UserRepository.class);

// CREATE
var user = User.builder()
    .name("Akhil").email("a@gmail.com").password("@$1469")
    .build();
repository.save(user);

// READ by ID
var user1 = repository.findById(1L).orElse(null);

// READ ALL
List<User> allUsers = repository.findAll();

// READ by Multiple IDs
List<Long> ids = new ArrayList<>();
ids.add(1L); ids.add(2L);
repository.findAllById(ids);

// DELETE
repository.deleteById(1L);

Managing Transactions

A transaction is a group of operations that either all succeed or all fail together. This keeps data consistent (e.g., create an order AND reduce stock — both or neither).

📄 UserService.java — @Transactional 
public class UserService {

    private final UserRepository userRepository;
    private final EntityManager entityManager; // access persistence context

    @Transactional  // All operations in this method = ONE transaction
    public void showEntityStates() {
        var user = User.builder()
            .name("John Doe").email("john@example.com")
            .password("password").build();

        // Check entity state
        if (entityManager.contains(user))
            System.out.println("Persistent");
        else
            System.out.println("Transient / Detached");

        userRepository.save(user); // Now it's Persistent
    }
}
@Transactional cannot be used on static methods! Transactions are method-level. When Lazy Loading throws LazyInitializationException, solve it by annotating the calling method with @Transactional to keep the persistence context open.

Persisting & Deleting Related Entities

📄 Cascade — Save Related Entities Automatically 
// Without CascadeType.PERSIST — must save address manually:
userRepository.save(user);
addressRepository.save(address); // extra line

// With CascadeType.PERSIST on @OneToMany — save user saves addresses too:
@OneToMany(mappedBy = "user", cascade = CascadeType.PERSIST)
private List<Address> addresses;

userRepository.save(user); // ✅ saves user AND all addresses
📄 Deleting Related Entities 
// For @OneToOne — add CascadeType.REMOVE to delete Profile with User
@OneToOne(mappedBy = "user", cascade = CascadeType.REMOVE)
private Profile profile;

// For child entity (orphanRemoval) — removes address when removed from list
@OneToMany(mappedBy = "user",
    cascade = {CascadeType.PERSIST, CascadeType.REMOVE},
    orphanRemoval = true)
private List<Address> addresses;

Derived Query Methods

Spring Data JPA can auto-generate SQL from method names in your repository interface. Follow the naming convention and Spring does the rest.

📄 ProductRepository.java — Derived Query Examples 
public interface ProductRepository extends CrudRepository<Product, Long> {

    // String matches
    List<Product> findByName(String name);                       // exact match
    List<Product> findByNameLike(String name);                   // LIKE (case-sensitive)
    List<Product> findByNameContaining(String name);             // contains
    List<Product> findByNameStartingWith(String name);           // starts with
    List<Product> findByNameEndingWithIgnoreCase(String name);   // case insensitive

    // Numbers
    List<Product> findByPriceGreaterThan(BigDecimal price);
    List<Product> findByPriceBetween(BigDecimal min, BigDecimal max);

    // Null checks
    List<Product> findByDescriptionNull();
    List<Product> findByDescriptionNotNull();

    // Multiple conditions
    List<Product> findByDescriptionNullAndNameNull();

    // Sort + Limit
    List<Product> findByNameOrderByPriceAsc(String name);

Custom Queries Using @Query

For complex queries, use @Query with either JPQL (Java Persistence Query Language) or native SQL.

💡
JPQL vs Native SQL JPQL uses class/field names (not table/column names). It works across all databases. Native SQL uses actual table names and database-specific syntax but exposes advanced SQL features.
📄 ProductRepository.java — @Query Examples 
// Native SQL Query (nativeQuery = true)
@Query(value = "SELECT * FROM products p WHERE p.price BETWEEN :min AND :max ORDER BY p.name",
       nativeQuery = true)
List<Product> findProducts(@Param("min") BigDecimal min,
                            @Param("max") BigDecimal max);

// JPQL Query (default — uses Model class name, not table)
@Query("SELECT p FROM Product p WHERE p.price BETWEEN :min AND :max ORDER BY p.name")
List<Product> findProductsJPQL(@Param("min") BigDecimal min,
                               @Param("max") BigDecimal max);

// JPQL COUNT
@Query("SELECT COUNT(*) FROM Product p WHERE p.price BETWEEN :min AND :max")
long countProducts(@Param("min") BigDecimal min, @Param("max") BigDecimal max);

// JPQL with JOIN
@Query("SELECT p FROM Product p JOIN p.category WHERE p.price BETWEEN :min AND :max")
List<Product> findProductsWithCategory(@Param("min") BigDecimal min,
                                          @Param("max") BigDecimal max);

// Update query — needs @Modifying + @Transactional
@Modifying
@Query("UPDATE Product p SET p.price = :newPrice WHERE p.category.id = :categoryId")
void updatePriceByCategory(BigDecimal newPrice, Byte categoryId);
Update/Delete Queries Need @Transactional Always add @Transactional in the calling service method when using @Modifying queries. Without it you get TransactionRequiredException.

Fetching Partial Data with Projections

Instead of loading all columns, use Projections to retrieve only the fields you need — better performance.

📄 Interface Projection — ProductSummary.java 
// Step 1: Create interface in 'dtos' or 'projections' package
public interface ProductSummary {
    Long getId();
    String getName();
}

// Step 2: Use as return type in repository
List<ProductSummary> findByCategory(Category category);

// With @Query:
@Query("SELECT p.id, p.name FROM Product p WHERE p.category = :category")
List<ProductSummary> findByCategory(@Param("category") Category category);
Interface vs Class Projections Prefer Interface projections — simpler, Spring generates proxy at runtime. Use Class (DTO) projections only when you need business logic in the projection.

Avoiding the N+1 Problem with @EntityGraph

With Lazy Loading, accessing N related objects triggers N+1 queries. Use @EntityGraph to load related data in a single JOIN query.

📄 UserRepository.java — @EntityGraph 
public interface UserRepository extends CrudRepository<User, Long> {

    // Load users WITH addresses in ONE query (prevents N+1)
    @EntityGraph(attributePaths = { "addresses" })
    @Query("SELECT u FROM User u")
    List<User> findAllWithAddresses();

    // Load with nested relations (addresses.country)
    @EntityGraph(attributePaths = { "addresses.country" })
    Optional<User> findByEmail(String email);
}

Dynamic Queries

When filter conditions change at runtime, write dynamic queries instead of dozens of static methods.

10.1 Query by Example

Filter using an example object. Simple way to filter data — Spring generates the SQL automatically based on non-null fields.

10.2 Criteria API

Programmatic query building using Java code. More complex but handles any filtering scenario.

10.3 Specifications API

Write reusable, composable query logic. Best for complex dynamic filters.

10.4 Sorting & Pagination

Retrieve results in sorted order and paginated chunks using Pageable and Sort.

📄 Query by Example (QBE) 
@Transactional
public void fetchProducts() {
    // Must use JpaRepository for findAll(Example)
    var product = new Product();
    product.setName("product"); // Search products containing "product"

    var matcher = ExampleMatcher.matching()
        .withIncludeNullValues()        // include null fields in match
        .withIgnorePaths("id", "description")   // ignore these fields
        .withStringMatcher(ExampleMatcher.StringMatcher.CONTAINING);

    var example = Example.of(product, matcher);
    var products = productRepository.findAll(example);
    products.forEach(System.out::println);
}
📄 Specifications API 
// Step 1: Extend JpaSpecificationExecutor
public interface ProductRepository extends
    JpaRepository<Product, Long>,
    JpaSpecificationExecutor<Product> {}

// Step 2: Create ProductSpec class
public class ProductSpec {
    public static Specification<Product> hasName(String name) {
        return (root, query, cb) ->
            cb.like(root.get("name"), "%" + name + "%");
    }

    public static Specification<Product> hasPriceBetween(BigDecimal min, BigDecimal max) {
        return (root, query, cb) ->
            cb.between(root.get("price"), min, max);
    }
}

// Step 3: Compose and use specs
var spec = ProductSpec.hasName("shoes")
    .and(ProductSpec.hasPriceBetween(
        BigDecimal.valueOf(10), BigDecimal.valueOf(100)));
var products = productRepository.findAll(spec);
📄 Sorting and Pagination 
// Sorting
var sort = Sort.by("name").ascending();
var products = productRepository.findAll(sort);

// Pagination
var pageable = PageRequest.of(0, 10, sort); // page 0, size 10, sorted by name
Page<Product> page = productRepository.findAll(pageable);

System.out.println(page.getTotalElements()); // total records
System.out.println(page.getTotalPages());    // total pages
page.getContent().forEach(System.out::println);

Calling Stored Procedures

When a query is too complex for JPQL/SQL, create a stored procedure in the database and call it from your repository.

📄 V10__add_find_products_proc.sql — Create Stored Procedure 
DELIMITER $$
CREATE PROCEDURE findProductsByPrice(
    minPrice DECIMAL(10,2),
    maxPrice DECIMAL(10,2)
)
BEGIN
    SELECT id, description, price, category_id
    FROM products
    WHERE price BETWEEN minPrice AND maxPrice
    ORDER BY name;
END $$
DELIMITER ;
📄 ProductRepository.java — @Procedure 
@Procedure("findProductsByPrice")
List<Product> findProducts(BigDecimal min, BigDecimal max);
📄 UserService.java — Calling Stored Procedure 
@Transactional  // Required for @Procedure — otherwise InvalidDataAccessException
public void fetchProducts() {
    var products = productRepository.findProducts(
        BigDecimal.valueOf(1),
        BigDecimal.valueOf(15)
    );
    products.forEach(System.out::println);
}
Always wrap in @Transactional Calling stored procedures via @Procedure requires a surrounding transaction. Without it you get InvalidDataAccessApiUsageException.
